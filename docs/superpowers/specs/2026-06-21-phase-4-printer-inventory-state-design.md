# Phase 4 Printer Inventory And State Design

## Goal

Phase 4 turns Phase 3 agent snapshots into tenant-scoped hub printer inventory and a read-only frontend state view. It does not implement print dispatch, authentication, printer credential storage, or a continuous agent-side MQTT subscription loop.

## Scope

- Persist printers reported by connected agents.
- Expose tenant-scoped HTTP APIs to list printers and request a refresh through the existing command ledger.
- Broadcast printer state changes over a tenant WebSocket.
- Show a minimal frontend printer inventory view backed by the hub API.
- Keep SQLite and PostgreSQL as first-class database backends.

Out of scope:

- User authentication and authorization beyond the existing tenant path boundary.
- Hub-managed printer credentials or assignment sync down to agents.
- Print/job dispatch.
- Live Bambu printer compatibility validation.
- A long-running agent MQTT report listener separate from explicit `RefreshPrinters`.

## Data Model

`pandar-core` gains a `Printer` domain record with:

- `id`: stable hub printer ID.
- `tenant_id`: tenant owner.
- `agent_id`: owning agent.
- `serial_number`: unique per tenant.
- `name`: display name from agent snapshot or serial fallback.
- `model`: optional model string when known.
- `status`: normalized machine state string.
- `last_seen_at`: latest snapshot timestamp.
- `created_at`: first registration timestamp.

The existing `printers` table already contains most columns. Phase 4 adds `last_seen_at TEXT` with backend-specific migrations. Existing rows are migrated with `last_seen_at = created_at` so repository reads never need a legacy fallback.

Printer IDs are generated by the hub as UUID strings. Upsert identity is `(tenant_id, serial_number)`. If the same serial is later reported by another agent in the same tenant, ownership moves to the latest reporting agent.

## Agent Snapshot Contract

The existing gRPC `PrinterSnapshot` event is the Phase 4 ingestion path. It keeps `serial`, `name`, and `state`, and adds optional `model`.

Hub handling rules:

- Only snapshots from the current live session token mutate state.
- Empty or malformed tenant/agent IDs continue to be rejected by the existing stream validation.
- Empty `serial`, `name`, or `state` in a snapshot is a stream `invalid_argument` error.
- `model` is trimmed and stored as `NULL` when empty.
- Snapshot timestamps use hub receive time, not agent-provided wall clock.
- Lower-level repository errors are logged with full context chains.

Agent rules:

- `MachineSnapshot` carries optional `model`.
- Snapshot events sent by `RefreshPrinters` include model from `PANDAR_PRINTERS` when configured.
- Empty printer config remains no-network and produces no printer snapshots.

## Hub Repository

`PrinterRepository` becomes a real repository, not just a count helper:

- `count()`.
- `list_for_tenant(tenant_id)`.
- `get_for_tenant(tenant_id, printer_id)`.
- `upsert_snapshot(tenant_id, agent_id, snapshot)`.

Repository behavior:

- Missing tenant on list returns `MissingTenant`.
- Missing agent on upsert returns `MissingAgent`.
- Unknown printer on get returns `Ok(None)`.
- Rehydrating invalid persisted data returns a repository error preserving the underlying core error context.
- SQLite and PostgreSQL queries are both covered by repository tests where the existing optional PostgreSQL harness can run.

## HTTP API

Phase 4 adds:

- `GET /api/v1/tenants/{tenant_id}/printers`
- `GET /api/v1/tenants/{tenant_id}/printers/{printer_id}`
- `POST /api/v1/tenants/{tenant_id}/agents/{agent_id}/refresh-printers`

Printer responses:

```json
{
  "id": "uuid",
  "tenant_id": "uuid",
  "agent_id": "uuid",
  "serial_number": "01S00EXAMPLE",
  "name": "garage-a1",
  "model": "A1 Mini",
  "status": "RUNNING",
  "last_seen_at": "2026-06-21T00:00:00Z",
  "created_at": "2026-06-21T00:00:00Z"
}
```

Refresh command response returns the created command record:

```json
{
  "id": "uuid",
  "tenant_id": "uuid",
  "agent_id": "uuid",
  "printer_id": null,
  "kind": "refresh_printers",
  "status": "queued",
  "payload_json": "{}",
  "error": null,
  "created_at": "2026-06-21T00:00:00Z",
  "updated_at": "2026-06-21T00:00:00Z"
}
```

Error mapping:

- Invalid tenant, agent, or printer IDs: `400`.
- Missing tenant, agent, or printer: `404`.
- Repository conflicts and command transition conflicts retain existing mappings.
- Database/internal errors return `500` and log full context.

## WebSocket API

Phase 4 adds:

- `GET /api/v1/tenants/{tenant_id}/printer-events`

The WebSocket validates the tenant ID and missing tenant before upgrade. It sends JSON messages for future events only; it does not replay historical state because HTTP listing is the source for initial state.

Message shape:

```json
{
  "type": "printer_snapshot",
  "printer": { "...": "PrinterResponse fields" }
}
```

Broadcast behavior:

- A tenant-scoped in-memory broadcaster lives in `AppState`.
- After a snapshot upsert succeeds, hub gRPC broadcasts the updated printer to that tenant.
- Slow or disconnected WebSocket clients may miss future events; clients should refresh over HTTP. This is acceptable for Phase 4 and avoids adding durable event storage.

## Frontend

The frontend becomes a small operational dashboard, not a landing page:

- Fetches `/api/v1/summary`.
- Fetches tenants and selects the first tenant by default.
- Fetches `/api/v1/tenants/{tenant_id}/printers`.
- Displays printer name, serial, model, status, agent ID, and last seen.
- Provides a tenant selector when more than one tenant exists.
- Shows empty and error states without requiring authentication.

The frontend does not open WebSockets in Phase 4. WebSocket support is validated at the hub boundary; frontend live subscription can be added later when authentication and tenant selection are stronger.

## Tests And Verification

Required tests:

- Core printer construction and validation.
- SQLite migration includes `last_seen_at`.
- Printer repository list/get/upsert behavior.
- Snapshot upsert moves ownership when the same serial is reported by a different agent in the same tenant.
- gRPC `PrinterSnapshot` persists state and ignores stale replaced streams.
- HTTP printer list/detail/refresh routes.
- WebSocket receives snapshot broadcast after an agent snapshot.
- Frontend TypeScript build/typecheck.

Required verification before commit:

- `cargo fmt`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo nextest run --manifest-path "Cargo.toml" --workspace`
- `npm run build` in `frontend/`
- `git diff --check`
- No protobuf generated Rust files outside `target`.

## Safety

- No real Bambu sockets are opened by Phase 4 tests.
- Hub WebSocket broadcasts are in-memory and tenant-scoped.
- Snapshot ingestion keeps stale-session protection from Phase 2.
- This phase adds no credential storage.
- Docs and roadmap must be updated with completed Phase 4 behavior and remaining Phase 5 handoff.
