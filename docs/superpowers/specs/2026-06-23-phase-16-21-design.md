# Phase 16-21 Product Completion Design

Phase 16-21 completes the next Pandar product slice after the Phase 15 live runtime dashboard. The work is one coordinated package because later UI, observability, artifact, and plugin paths depend on Phase 16 tenant-owned credentials and authenticated agent sessions.

The implementation must stay incremental and testable. It must not introduce a full Bambu cloud clone, a slicer parser, production packaging for the network plugin, or compatibility fallbacks for old user-owned API tokens.

## Current State

- `pandar-hub` authenticates bearer API tokens through `api_tokens` joined to `users`; token capability inherits the current user's `role`.
- Phase 11 provisioning APIs can create users, user-owned API tokens, identity links, and agent pairing bundles.
- Agent pairing bundles return `PANDAR_TENANT_ID`, `PANDAR_AGENT_ID`, and `PANDAR_AGENT_NAME`; gRPC reverse connections trust those IDs in the first `AgentHello` event.
- The frontend is a tenant runtime dashboard with dispatch, discovery, diagnostics, material/job display, and browser-safe WebSocket updates.
- Existing backend routes include refresh/discover/diagnose command enqueueing and print-job creation, but the UI does not expose tenant-admin workflows, manual refresh, safe reprint, health/metrics, or plugin login.
- `reference/open-bamboo-networking` exists for Phase 21 ABI compatibility, and `reference/BambuStudio` exists for caller behavior.

## Goals

1. Replace user-owned API tokens with tenant-owned scoped tokens for all bearer API automation.
2. Add authenticated agent enrollment and credential rotation so reverse gRPC sessions are bound to hashed agent credentials, not trusted request IDs.
3. Expose tenant-admin provisioning, tenant tokens, identity links, agent pairing, and audit history in the frontend without moving authorization out of Rust.
4. Add typed operator controls and recovery flows for refresh, discovery, diagnostics, unavailable pause/resume/stop capability indicators, failed dispatch retry, and explicit reprint.
5. Add operational health, readiness, metrics, redaction coverage, retention cleanup, and backup guidance for self-hosting.
6. Improve artifact submission and reprint ergonomics without parsing slicer internals in hub core.
7. Scaffold a minimal `pandar-network-plugin` `cdylib` with ABI export coverage, hub-only login-ticket exchange, and documented install paths.

## Non-Goals

- No acceptance of legacy `api_tokens` for bearer API authentication after Phase 16 migrations and route updates.
- No Bambu access codes, LAN addresses, or printer MQTT/FTPS logic in the network plugin.
- No arbitrary slicer-file parser in hub core.
- No production installer or signed package generation for the plugin.
- No browser-side persistence of plaintext tenant tokens, agent credentials, plugin tickets, or plugin session credentials after copy/exchange.
- No full clone of Bambu cloud APIs beyond the minimal ABI and hub calls needed for Studio load/login/status/print paths.

## Phase 16: Tenant Tokens And Agent Enrollment

### Data Model

Add backend-neutral SQLite and PostgreSQL migrations for:

- `tenant_tokens`
  - `id`, `tenant_id`, `name`, `token_hash`, `scopes_json`, `created_by_user_id`, `created_at`, `last_used_at`, `expires_at`, `revoked_at`.
  - Belongs directly to `tenant_id`.
  - `created_by_user_id` is nullable audit metadata and never used for authorization.
  - Plaintext values use a generated `pandar_tenant_` prefix for ordinary tenant tokens and a generated `pandar_plugin_` prefix for plugin-session tokens.
- `plugin_login_tickets`
  - `id`, `tenant_id`, `user_id`, `ticket_hash`, `redirect_url`, `created_at`, `expires_at`, `used_at`, `revoked_at`.
  - Tickets are one-use, expire after 5 minutes by default, and are exchanged only for plugin-scoped tenant tokens.
- Agent credential columns on `agents`
  - `credential_hash`, `credential_rotated_at`, `credential_revoked_at`.
  - `credential_hash` is nullable in the migration so existing rows migrate. Under the no-legacy-fallback rule, agents with null `credential_hash` cannot authenticate new gRPC sessions and must be re-paired or rotated through the Phase 16 APIs.

SeaORM entities and repositories must cover both backends with parity tests. Plaintext token and credential values are returned once, hashed before persistence, and never listed later. Hashes use the existing repository SHA-256 helper over the full plaintext secret. Plaintext random segments for tenant tokens, plugin tokens, plugin login tickets, and agent credentials must contain at least 16 bytes from a cryptographically secure random source such as `getrandom`, encoded with a URL-safe alphabet. If UUIDv4 is used instead, use at least two independent UUIDv4 values per secret so the random portion exceeds 128 bits.

### Scope Semantics

Tenant-token scopes are the only capability source for bearer automation:

- Empty `scopes` grants read-only tenant access equivalent to viewer routes.
- `["*"]` grants all tenant-scoped API capabilities and agent registration.
- `["agent:register"]` grants only agent registration, pairing, and agent credential rotation endpoints.
- `["plugin:studio"]` grants Bambu Studio plugin access to read tenant printers/jobs and submit print jobs through plugin routes only. It does not grant tenant-admin APIs or `agent:register`.

External identity bearer auth continues to resolve Pandar-owned users and roles. Tenant tokens and external users share an authorization interface, but tenant tokens do not inherit user roles.

External identity bearer auth means the configured Clerk/Logto-compatible JWT verifier from Phase 10: `Authorization: Bearer <provider JWT>` is verified against `PANDAR_EXTERNAL_AUTH_*`, then mapped through `user_identities` to a local Pandar user and role. Removed user-owned `api_tokens` are Pandar-minted database tokens and are distinct from provider JWTs. After Phase 16, accepted bearer values are only tenant tokens, plugin-scoped tenant tokens on plugin routes, bootstrap tokens on bootstrap-only routes, and configured external provider JWTs when external auth is enabled.

Scope arrays are additive except that `["*"]` supersedes all other tenant scopes. The implementation must reject unknown scopes on tenant-token creation and rotation.

### Audit Actors

New token-authenticated mutations must use the existing audit event schema:

- External user actions: `actor_type = "user"`, `user_id = Some(user_id)`.
- Tenant-token actions: `actor_type = "tenant_token"`, `user_id = created_by_user_id` from the token when present, and metadata must include `{ "tenant_token_id": "...", "tenant_token_scopes": [...] }`.
- Plugin-token actions: `actor_type = "plugin_token"`, `user_id = created_by_user_id` from the token when present, and metadata must include `{ "tenant_token_id": "...", "tenant_token_scopes": ["plugin:studio"] }`.
- Bootstrap actions keep the existing bootstrap actor behavior.

Audit metadata must never include plaintext tokens, token hashes, agent credentials, plugin tickets, Bambu access codes, artifact storage paths, external provider subjects, or bearer headers.

### APIs

Tenant-admin APIs must support:

- `GET /api/v1/tenants/{tenant_id}/tenant-tokens`
- `POST /api/v1/tenants/{tenant_id}/tenant-tokens`
- `DELETE /api/v1/tenants/{tenant_id}/tenant-tokens/{token_id}`
- `POST /api/v1/tenants/{tenant_id}/tenant-tokens/{token_id}/rotate`

Tenant-token API contracts:

- `GET /api/v1/tenants/{tenant_id}/tenant-tokens`
  - Auth: tenant admin external user or `*` tenant token.
  - Response `200`: `{ "tenant_tokens": [TenantTokenResponse] }`.
- `POST /api/v1/tenants/{tenant_id}/tenant-tokens`
  - Auth: tenant admin external user or `*` tenant token.
  - Request: `{ "name": string, "scopes": string[], "expires_at": string | null }`.
  - `name` must be non-empty after trimming. `scopes` is required and may be an empty array. `expires_at` must be RFC3339 when present.
  - Response `201`: `{ "tenant_token": TenantTokenResponse, "token": "pandar_tenant_<random>" }`.
  - Errors: `400 bad_request`, `400 invalid_scope`, `400 invalid_expires_at`, `401/403` auth errors.
- `DELETE /api/v1/tenants/{tenant_id}/tenant-tokens/{token_id}`
  - Auth: tenant admin external user or `*` tenant token.
  - Response `200`: `{ "tenant_token": TenantTokenResponse }` with `revoked_at` set.
  - Revocation is idempotent for an existing tenant token.
- `POST /api/v1/tenants/{tenant_id}/tenant-tokens/{token_id}/rotate`
  - Auth: tenant admin external user or `*` tenant token.
  - Request: `{ "expires_at": string | null }`; `name` and `scopes` are copied from the old token.
  - Behavior: revoke the old token and create a new token in one transaction.
  - Response `201`: `{ "tenant_token": TenantTokenResponse, "token": "pandar_tenant_<random>", "rotated_from_token_id": string }`.

`TenantTokenResponse` contains `id`, `tenant_id`, `name`, `scopes`, `created_by_user_id`, `created_at`, `last_used_at`, `expires_at`, and `revoked_at`. It never contains `token_hash` or plaintext token values.

Agent registration APIs must support tenant tokens with `agent:register` or `*`:

- `POST /api/v1/tenants/{tenant_id}/agent-pairings` returns `PANDAR_TENANT_ID`, `PANDAR_AGENT_ID`, `PANDAR_AGENT_NAME`, and `PANDAR_AGENT_CREDENTIAL`.
- `POST /api/v1/tenants/{tenant_id}/agents/{agent_id}/credential:rotate` returns a new plaintext agent credential once.
- `POST /api/v1/tenants/{tenant_id}/agents/{agent_id}/credential:revoke` sets `credential_revoked_at`, prevents new gRPC sessions, closes the current in-memory session when present, and returns the public `AgentResponse`.
- Revoking or rotating an agent credential prevents new reverse gRPC sessions with the old credential.

The old user-scoped API-token routes must return `410 Gone` and must not mint credentials accepted by bearer API auth. The retired routes are:

- `GET /api/v1/tenants/{tenant_id}/users/{user_id}/api-tokens`
- `POST /api/v1/tenants/{tenant_id}/users/{user_id}/api-tokens`
- `DELETE /api/v1/tenants/{tenant_id}/api-tokens/{token_id}`

Each retired route returns `{ "error": "api_tokens_retired" }` with status `410 Gone`, regardless of bearer principal. Bootstrap tenant-admin creation must return a tenant token under the Phase 16 model instead of a user-owned API token.

Audit-history API contract:

- `GET /api/v1/tenants/{tenant_id}/audit-events`
  - Auth: tenant admin external user or `*` tenant token.
  - Query: `limit` optional integer `1..=100`, default `50`; `before` optional RFC3339 timestamp; `action` optional exact action string.
  - Response `200`: `{ "audit_events": [AuditEventResponse] }`.
  - `AuditEventResponse`: `{ "id": string, "tenant_id": string, "actor_type": string, "user_id": string | null, "action": string, "target_type": string, "target_id": string | null, "metadata": object, "created_at": string }`.
  - Results sort newest first. `metadata` is parsed from `metadata_json`; invalid persisted JSON is returned as `{}` and logged with full error context.
  - Response metadata must be redacted with the same forbidden fields as audit write metadata.

### gRPC Authentication

Update `proto/pandar/agent/v1/agent.proto` so `AgentHello` includes an agent credential. `AgentEvent.tenant_id` remains in the envelope for routing and tests, and `PANDAR_TENANT_ID` remains in the pairing bundle so the agent can populate that envelope. It is not a secret and is never sufficient for authentication.

Initial gRPC authentication is:

1. Parse `AgentEvent.tenant_id` and `AgentEvent.agent_id`.
2. Load the agent by `agent_id`.
3. Reject with `not_found` if no agent exists.
4. Reject with `permission_denied` if the loaded agent's `tenant_id` does not equal the event `tenant_id`.
5. Reject with `unauthenticated` if `AgentHello.credential` is missing, hashes to a value different from `agents.credential_hash`, or the credential is revoked.
6. Accept only after the credential hash matches, then mark the agent online and register the session for the loaded `(tenant_id, agent_id)`.

All later inbound gRPC events remain bound to the authenticated `(tenant_id, agent_id)` from the accepted session. If an event carries mismatched IDs, it is rejected or ignored according to existing stale-session behavior. Existing replacement-session protection through `SessionToken` remains.

`pandar-agent` CLI/env config must replace manual tenant trust with:

- `PANDAR_TENANT_ID`
- `PANDAR_AGENT_ID`
- `PANDAR_AGENT_CREDENTIAL`
- `PANDAR_HUB_GRPC_URL`
- existing agent name/version/printer/artifact config

The agent includes the credential only in `AgentHello`. Heartbeats, command acknowledgements, command results, snapshots, and print reports do not repeat the credential.

### Acceptance Criteria

- Old user-owned API tokens are no longer accepted for bearer API authentication.
- Tenant tokens list without plaintext values and record last use.
- Empty-scope tokens can read tenant resources and cannot mutate them.
- `*` tokens can perform tenant-scoped mutations.
- `agent:register` tokens can create/rotate agent credentials and cannot read ordinary tenant resources.
- Revoked or rotated tenant tokens and agent credentials fail subsequent protected API/gRPC access.
- Existing command dispatch, heartbeat, snapshot, print report, and command result handling prove authenticated agent identity is enforced.

## Phase 17: Tenant Admin Product UI

Add frontend admin sections to the existing tenant dashboard instead of introducing a separate product shell:

- Tenant users: list users, create user, change role.
- Identity links: list and link provider subjects for a selected user.
- Tenant tokens: list tenant-owned tokens, create scoped tokens, revoke tokens, rotate tokens.
- Agent pairing: create a pairing bundle and show the generated env block once.
- Audit history: read-only provisioning audit events.

The UI must use Rust authorization decisions. It may hide admin sections when requests return forbidden, but must not duplicate role policy as the source of truth.

Copy-once secret flows must render plaintext values only in the creation/rotation response state and must not persist them to Zustand, local storage, cookies, URL query strings, or server-rendered cached state.

### Acceptance Criteria

- Tenant admins can onboard an operator/viewer, link an external subject, create/revoke/rotate tenant tokens, and pair an agent from the product UI.
- Viewer/operator users cannot complete tenant-admin mutations through the UI or API.
- Plaintext secrets are shown only once after creation/rotation.

## Phase 18: Command Controls And Recovery UX

Expose existing controls in the runtime dashboard and add recovery flows. This package does not implement backend pause, resume, or stop commands; those controls are rendered as unavailable capability indicators only.

- Manual refresh button for each linked agent.
- Discovery and diagnostics remain visible and show the latest command result inline.
- Failed dispatch retry creates a new command/job only from a failed pre-physical-dispatch state and labels the new attempt explicitly.
- Reprint duplicates a completed job's artifact and print settings through an explicit operator action.
- Pause/resume/stop controls are not clickable in this package. They show unavailable state with the specific missing capability from compatibility/diagnostics data.
- Timeout/stale-agent messaging must distinguish hub enqueue, agent offline, file transfer, MQTT publish, and physical print state.

Recovery API contracts:

- `POST /api/v1/tenants/{tenant_id}/jobs/{job_id}/retry-dispatch`
  - Auth: operator+ external user or `*` tenant token.
  - Eligible source: job `status = failed`, command status `failed`, `print.status = pending`, `print.started_at` is null, `print.progress_percent` is null or `0`, and `print.current_layer` is null or `0`. This means no physical print start/progress was reported.
  - Ineligible source: command status `queued`, `sent`, `acknowledged`, or `succeeded`; job status other than `failed`; physical print status `running`, `completed`, `failed`, or `cancelled`; any non-null `print.started_at`; any positive progress/layer evidence. Ineligible requests return `409 retry_not_safe`.
  - Request: `{ "reason": string | null }`.
  - Behavior: creates a new command for the same job/artifact/printer settings, keeps the original job record, sets the job dispatch status back to queued/running according to the existing command enqueue pattern, and writes audit action `job.retry_dispatch`.
  - Response `201`: existing `JobResponse` shape with the new `command_id`.
- `POST /api/v1/tenants/{tenant_id}/jobs/{job_id}/reprint`
  - Auth: operator+ external user or `*` tenant token.
  - Eligible source: any job with an existing artifact and terminal physical print state `completed`, `failed`, or `cancelled`.
  - Request: `{ "reason": string | null }`.
  - Behavior: creates a new job row and command row that reuse the stored artifact metadata and print settings. It does not trust browser-supplied storage paths. Audit action: `job.reprint`.
  - Response `201`: `JobResponse` for the new job.
- `POST /api/v1/tenants/{tenant_id}/jobs/{job_id}/duplicate`
  - Auth: operator+ external user or `*` tenant token.
  - Eligible source: any job with an existing artifact, including a currently running or non-terminal job. Duplicate creates an independent queued follow-up and never mutates or stops the source job.
  - Request: `{ "printer_id": string | null, "plate_id": number | null, "use_ams": boolean | null, "flow_cali": boolean | null, "timelapse": boolean | null, "ams_mapping": object | null, "ams_mapping2": object | null }`.
  - Behavior: creates and enqueues a new print job using copied settings, overridden only by non-null request fields. Audit action: `job.duplicate`.
  - Response `201`: `JobResponse` for the new job.

The UI must label this action "Duplicate and print" because Phase 20 does not add a non-dispatching draft job state.

### Acceptance Criteria

- Operators can refresh, discover, diagnose, inspect command results, retry safe failed dispatches, and reprint from the dashboard.
- Recovery actions create audit records and preserve dispatch-vs-physical-print separation.
- The UI never suggests that a dispatch success means physical print completion.

## Phase 19: Operational Reliability And Observability

Add operations endpoints and docs for self-hosted deployments:

- `GET /healthz` remains a shallow liveness endpoint.
- `GET /readyz` checks database access, gRPC listener configuration, artifact spool directory access, and external auth configuration validity. If external auth is disabled, `external_auth` is ready (`1`) and the JSON detail is `"disabled"` because disabled external auth is an intentional deployment mode, not a readiness failure.
- `GET /metrics` returns Prometheus text metrics for agent sessions, command lifecycle counts, WebSocket subscriptions, ticket issuance, job outcomes, print-report ingestion, and runtime health counters.
- Add a `pandar-app cleanup` CLI command. It supports `--dry-run` and `--execute`, defaults to dry-run, and reads the same `PANDAR_DATABASE_URL` and artifact root settings as the hub. It does not run automatically in the hub process.
- Cleanup settings:
  - `PANDAR_RETENTION_COMPLETED_JOBS_DAYS`, default `90`.
  - `PANDAR_RETENTION_COMMANDS_DAYS`, default `90`.
  - `PANDAR_RETENTION_MACHINE_EVENTS_DAYS`, default `30`.
  - `PANDAR_RETENTION_AUDIT_DAYS`, default `365`.
  - `PANDAR_RETENTION_EXPIRED_TICKETS_DAYS`, default `7`.
  - `PANDAR_RETENTION_REVOKED_TOKENS_DAYS`, default `365`.
- Cleanup targets are completed/failed/cancelled jobs and their unreferenced artifacts, terminal old commands, old machine events, old audit records, expired/used plugin login tickets, and revoked/expired tenant tokens.
- Cleanup must exclude active jobs, queued/running commands, current artifacts referenced by active or retained jobs, active tenant tokens, active agent credentials, and non-terminal records.
- `--dry-run` reports counts and artifact bytes without deleting. `--execute` deletes inside backend-specific transaction boundaries where database rows are involved, then removes unreferenced artifact files. The Phase 19 cleanup CLI emits structured logs only and does not write audit events; API-triggered cleanup audit logging is deferred outside this package.
- Add log redaction tests for bearer tokens, WebSocket tickets, Bambu access codes, agent credentials, plugin tickets, and artifact paths.
- Document SQLite/PostgreSQL backup and restore commands.

Metrics contract:

- `pandar_agent_sessions{state="online|offline"}` gauge, counted from session registry and persisted agent status only.
- `pandar_commands_total{kind,status}` counter or monotonic scrape value derived from command rows. Labels are command kind and normalized status only.
- `pandar_websocket_subscriptions{tenant_id_hash}` gauge. The label is a stable hash of tenant id, not the raw tenant id.
- `pandar_websocket_tickets_total{result="issued|consumed|expired|invalid"}` counter.
- `pandar_jobs_total{status,print_status}` counter or monotonic scrape value derived from job rows.
- `pandar_print_reports_total{result="accepted|rejected"}` counter.
- `pandar_readyz{check="database|grpc|spool|external_auth"}` gauge, `1` for ready and `0` for failing. Disabled external auth reports `external_auth = 1`.
- `tenant_id_hash` is the first 16 lowercase hex characters of SHA-256 over the tenant id string.
- Forbidden metric labels/values: raw tenant ids, user ids, token ids, bearer values, token hashes, WebSocket tickets, agent credentials, Bambu serials, Bambu access codes, artifact filenames, artifact paths, external provider subjects, and plugin tickets.

Cleanup selection rules:

- Jobs: select terminal jobs where `updated_at` is older than `PANDAR_RETENTION_COMPLETED_JOBS_DAYS`; terminal means job dispatch status is `succeeded` or `failed` and physical print status is `completed`, `failed`, `cancelled`, or `pending` with no queued/sent/acknowledged command. Do not delete any job whose command is queued/sent/acknowledged or whose physical print status is running.
- Artifacts: after selecting jobs for deletion, delete artifact rows and files only when no remaining job references the artifact id. Artifact file deletion happens after the database transaction commits; failed file deletion is logged with full context and leaves no extra database mutation.
- Commands: delete terminal commands older than `PANDAR_RETENTION_COMMANDS_DAYS` only when no retained job references the command id. Terminal means succeeded or failed.
- Machine events: delete rows older than `PANDAR_RETENTION_MACHINE_EVENTS_DAYS`.
- Audit events: delete rows older than `PANDAR_RETENTION_AUDIT_DAYS`; never delete audit events newer than retained jobs/commands they describe if a direct target id relation is available.
- Plugin login tickets: delete used, revoked, or expired tickets older than `PANDAR_RETENTION_EXPIRED_TICKETS_DAYS`.
- Tenant tokens: delete revoked or expired tenant-token rows older than `PANDAR_RETENTION_REVOKED_TOKENS_DAYS`; never delete active unexpired tokens.
- SQLite and PostgreSQL implementations must run database deletions in one transaction per cleanup category to keep locks bounded. Dry-run performs the same selection queries without mutation. Tests must cover SQLite always and PostgreSQL when `PANDAR_TEST_POSTGRES_URL` is configured.

### Acceptance Criteria

- Operators can distinguish app, database, agent, and printer failures from readiness, metrics, and logs.
- Disabled external auth reports ready in `/readyz` and `pandar_readyz{check="external_auth"}`; invalid configured external auth reports not ready.
- Sensitive credentials are redacted in logs and metrics.
- Cleanup behavior is explicit, test-covered, and does not delete active jobs or current artifacts.

## Phase 20: Artifact And Slicer Workflow Polish

Improve the current print submission flow while keeping slicer parsing out of hub core:

- Client-side upload UX shows selected file name, size, base64 conversion progress, and the configured max artifact size.
- The configured max artifact size is `PANDAR_MAX_ARTIFACT_BYTES`, default `268435456` bytes.
- Backend validation returns stable error codes:
  - `400 artifact_empty` when decoded artifact bytes are empty.
  - `400 artifact_invalid_base64` when the submitted artifact payload is not valid base64.
  - `400 artifact_invalid_plate` when `plate_id` is negative or cannot be represented by the printer command model.
  - `413 artifact_too_large` when decoded bytes exceed `PANDAR_MAX_ARTIFACT_BYTES`.
  - `404 printer_not_found` when the selected printer does not belong to the tenant or does not exist.
- Phase 20 does not introduce additional artifact-level backend endpoints. It adds frontend UX and backend validation on the existing print submission route plus the Phase 18 `reprint` and `duplicate` job actions.
- Job responses already expose artifact metadata; frontend actions must call Phase 18 `reprint` and `duplicate` to reuse existing artifacts safely.
- Material mapping helpers in the frontend may derive suggestions from current printer material snapshots, but backend validation remains authoritative.
- Preserve filename, content type, size, storage path, plate, print flags, and material mapping as operator-inspectable job metadata.

### Acceptance Criteria

- Operators can upload, inspect, duplicate, and reprint project artifacts through the UI.
- Reprint/duplicate flows reuse stored artifacts without trusting browser-supplied storage paths.
- Material mapping stays explicit and validated.

## Phase 21: Bambu Studio Network Plugin

Add `crates/pandar-network-plugin` as a Rust `cdylib` crate in the workspace.

The approved ABI implementation path is a small C++17 shim compiled from the plugin crate build script and linked into the Rust `cdylib`. The shim exports the exact Bambu Studio C++ ABI symbols that use `std::string`, callback function types, and `ft_*` structs. The shim delegates hub-backed work to Rust through a narrow `extern "C"` boundary that uses raw UTF-8 pointers, lengths, integer status codes, and opaque handles only. This avoids trying to expose C++ ABI types directly from Rust.

Build constraints:

- Add `cc` as a build dependency for `crates/pandar-network-plugin` if no existing build tool can compile the shim.
- The shim must compile as C++17 on Linux, macOS, and Windows toolchains supported by Cargo.
- C++ owns C++ ABI return values such as `std::string` by value. Rust never allocates or frees `std::string`.
- Rust-owned strings returned to C++ through the internal C boundary use explicit pointer/length structs plus a Rust free function, or are copied into C++ `std::string` before returning to Studio.
- `ft_*` handles are opaque C++ shim structs in Phase 21. Retain/release counters live in the shim, and Rust does not inspect their layout.

### ABI

Use `reference/open-bamboo-networking` and `reference/BambuStudio` as references. The normative export list for this phase is `docs/superpowers/specs/2026-06-23-phase-21-network-plugin-abi-symbols.txt`, copied from `reference/open-bamboo-networking/tests/probe_plugin.cpp`. The implementation must export every symbol in that local file, including the `ft_*` symbols, with ABI-compatible signatures derived from the same reference files.

Minimal behavior is:

- Lifecycle, callback registration, metadata, user/login, connection, printer/job, settings, tracking, camera, MakerWorld, and filament symbols return stable empty/default success or unsupported responses unless this spec assigns a hub-backed behavior.
- `bambu_network_get_bambulab_host`, `bambu_network_get_my_token`, `bambu_network_get_my_profile`, user login state, printer reads, job reads, and print submission are hub-backed.
- `ft_abi_version` returns `1`; other `ft_*` functions keep Studio from crashing and return explicit failed/no-op results without opening Bambu LAN sockets.
- No symbol opens direct agent, MQTT, FTPS, SFTP, or Bambu LAN connections.

### Login And Hub Access

Add hub/frontend login-ticket flow:

- `bambu_network_get_bambulab_host` returns the configured Pandar frontend URL.
- The frontend serves a Studio-compatible sign-in entry page.
- The page lets the user confirm or enter the Pandar URL when needed, then uses the configured external auth flow.
- If external auth is disabled, the sign-in entry page returns a visible configuration error and does not create plugin login tickets.
- After authentication and tenant selection, the frontend creates a short-lived one-use plugin login ticket.
- The browser redirects to Studio's localhost callback with `ticket` and `redirect_url` using Studio's `get_localhost_url` message when available.
- Studio calls plugin `get_my_token(ticket)` and `get_my_profile(token)`.
- The plugin exchanges the ticket at `POST /api/v1/plugin/login-tickets/exchange` and receives a plaintext `pandar_plugin_` token once plus profile JSON.
- The exchange hashes the ticket before lookup, requires `used_at` and `revoked_at` to be null and `expires_at` to be in the future, marks it used in the same transaction that creates the plugin token, and fails replay attempts.
- The resulting hub credential is a `tenant_tokens` row with `["plugin:studio"]`, a default 30-day expiration, nullable `created_by_user_id` copied from the login ticket, and normal revocation/expiry behavior. It is not a user-owned API token, not empty-scope read-only, not `*`, and not `agent:register`.
- `get_my_profile(token)` returns Bambu-shaped user/tenant profile JSON derived from the token's tenant and creator metadata; it must not reveal tenant tokens, agent credentials, Bambu access codes, or local artifact paths.

### Printer And Job Calls

The plugin maps Studio network calls to hub APIs:

- read tenant printers and jobs from hub;
- submit print jobs through hub APIs;
- consume or cache hub live status where synchronous ABI calls need a snapshot;
- return contextual errors for plugin login, hub API, agent offline, and printer runtime failures.

### Plugin HTTP Contracts

Ticket creation:

- `POST /api/v1/tenants/{tenant_id}/plugin/login-tickets`
- Auth: viewer+ external user for the selected tenant or `*` tenant token. Empty-scope tenant tokens are denied because this endpoint mints credentials.
- Request: `{ "redirect_url": string }`.
- `redirect_url` must parse as an absolute URL with scheme `http`, no username, no password, no fragment, and host exactly `localhost`, `127.0.0.1`, or `[::1]`. The port must be present and in `1..=65535`. The path and query are preserved and returned unchanged. Any other scheme, host, missing port, credentials, fragment, or parse failure returns `400 invalid_redirect_url`.
- Response `201`: `{ "ticket": "pandar_plugin_ticket_<random>", "expires_at": string, "redirect_url": string }`.
- The plaintext ticket is returned once; only `ticket_hash` is stored.

Ticket exchange:

- `POST /api/v1/plugin/login-tickets/exchange`
- Auth: no bearer header; possession of the one-use ticket is the credential.
- Request: `{ "ticket": string }`.
- Response `200`: `{ "token": "pandar_plugin_<random>", "expires_at": string, "profile": { "user_id": string, "user_name": string, "tenant_id": string, "tenant_name": string } }`.
- Replay, expired, revoked, malformed, or unknown tickets return `401 invalid_plugin_ticket`.
- A successful exchange marks the ticket used and creates the `["plugin:studio"]` tenant token in one transaction.

Plugin route wrappers:

- `GET /api/v1/plugin/printers`
  - Auth: `plugin:studio` token.
  - Response `200`: `{ "printers": [PluginPrinter] }`.
  - `PluginPrinter`: `{ "dev_id": string, "name": string, "model": string | null, "online": boolean, "state": string, "pandar_printer_id": string }`.
  - `dev_id` is the Pandar printer id in Phase 21; it is not the Bambu serial. Bambu serials, LAN addresses, and access codes are excluded.
- `GET /api/v1/plugin/jobs`
  - Auth: `plugin:studio` token.
  - Response `200`: `{ "jobs": [PluginJob] }`.
  - `PluginJob`: `{ "task_id": string, "dev_id": string, "name": string, "status": string, "progress_percent": number | null, "created_at": string, "updated_at": string, "pandar_job_id": string }`.
  - `name` is the artifact filename only. Local artifact storage paths are excluded.
- `POST /api/v1/plugin/prints`
  - Auth: `plugin:studio` token.
  - Request: `{ "printer_id": string, "filename": string, "content_type": string, "artifact_base64": string, "plate_id": number, "use_ams": boolean, "flow_cali": boolean, "timelapse": boolean, "ams_mapping": object | null, "ams_mapping2": object | null }`.
  - Response `201`: `{ "task_id": string, "command_id": string, "status": string, "message": string | null, "pandar_job_id": string }`.
  - Validation delegates to the same backend rules as ordinary print-job creation.
  - Plugin route errors use `{ "error": string, "message": string | null }` with existing HTTP status codes. Error messages must not include secrets, Bambu access codes, or artifact storage paths.

Hub-backed ABI behavior:

- `bambu_network_get_bambulab_host` returns the configured frontend URL string.
- `bambu_network_get_my_token` exchanges the one-use ticket and returns the plugin token JSON string from `POST /api/v1/plugin/login-tickets/exchange`; exchange failures return a non-zero ABI code and store a redacted last-error string.
- `bambu_network_get_my_profile` returns the `profile` object from the exchange response or a profile fetched by plugin token if already exchanged.
- User login state is true only while a non-expired plugin token is present in plugin memory.
- Printer/job/read/print ABI calls translate to the plugin route wrappers above.
- Callback registration symbols store callback pointers but do not call them until a hub-backed event needs it. Missing callbacks are not errors.
- Unsupported ABI calls return stable empty/default payloads or non-zero unsupported codes without panic.

## Rollback And Safety

The migration intentionally removes bearer acceptance for old user-owned API tokens. Rollback requires restoring the previous schema/code from git and issuing fresh accepted credentials; the implementation must not include a hidden compatibility path.

Secrets must remain one-time display values. Any log or metric that includes bearer credentials, WebSocket tickets, Bambu access codes, agent credentials, plugin tickets, or artifact filesystem paths is a blocking bug.

New hub, agent, plugin, CLI, and frontend server-action error paths must preserve lower-level cause chains at module/runtime boundaries. Rust logs for `anyhow` errors must use `{err:#}` or equivalent full-chain formatting, and handlers must not replace external-system, database, gRPC, filesystem, config, or network errors with context-free strings.
