# Phase 25 Scaled Artifact Storage And Upload Pipeline Design

## Objective

Remove the shared local spool directory as the limiting factor for horizontally scaled print-job creation. Phase 25 introduces an explicit artifact-storage boundary, keeps filesystem storage as the SQLite/single-node default, adds an object-storage backend for PostgreSQL plus multi-Hub deployments, and replaces browser/plugin base64 job creation with streaming upload paths that preserve the existing job validation and error labels.

The hub must still treat slicer files as opaque artifacts. This phase changes storage, retrieval, upload transport, cleanup, and operational documentation; it does not parse 3MF/G-code contents or infer slicer metadata.

## Current State

- `pandar-hub` uses `JobStorageConfig` in `crates/pandar-hub/src/jobs.rs` to write uploaded bytes under `PANDAR_SPOOL_DIR`, defaulting to `pandar-spool`.
- Browser job creation posts JSON with `artifact_base64` through the Next.js server action in `frontend/app/actions.ts`.
- The network plugin reads the local artifact file and posts JSON with `artifact_base64` to `POST /api/v1/plugin/prints`.
- Job artifact metadata is already stored in `job_artifacts` for both SQLite and PostgreSQL: filename, content type, size, and `storage_path`.
- Reprint and duplicate create new job/command rows from the existing artifact row and do not accept browser-supplied storage paths.
- Cleanup discovers stale artifact rows from the database, removes files with `JobStorageConfig::remove_artifact`, then deletes artifact rows.
- `/readyz` and Prometheus expose a `spool` check tied to local filesystem access.
- `PrintProjectFile` gRPC commands only carry `storage_path` as the artifact byte locator; `pandar-agent` reads bytes from `PANDAR_ARTIFACT_ROOT + storage_path`. PostgreSQL plus NATS deployments therefore still need a shared POSIX spool volume.

## Storage Contract

Introduce a hub-side `ArtifactStorage` boundary with this behavior:

- `put_artifact(tenant_id, artifact_id, filename, stream/bytes) -> StoredArtifact`
- `open_artifact(tenant_id, artifact_id, storage_key) -> stream/bytes`
- `delete_artifact(storage_key) -> deleted-or-not-found`
- `check_ready() -> storage health`

`StoredArtifact` must include sanitized filename, opaque storage key, size in bytes, and a backend identifier. Existing API responses must continue to omit storage paths. Internal logs must preserve cause chains while redacting local paths, object keys, bucket names when sensitive, credentials, and signed URL material.

Storage keys are hub-generated. Browser, plugin, and agent requests must never supply a storage key for a new artifact. Existing `job_artifacts.storage_path` can continue to hold the hub-generated opaque key for Phase 25 to avoid broad schema churn, but public docs and code comments should stop describing it as a local filesystem path. If a backend identifier is added to the database, it must be migrated for both SQLite and PostgreSQL with the same external behavior.

### Filesystem Backend

The filesystem backend preserves today’s simple deployment:

- Default backend for SQLite and single-node deployments.
- Uses `PANDAR_SPOOL_DIR` and `PANDAR_MAX_ARTIFACT_BYTES`.
- Keeps current relative-path safety rules for delete/read.
- Can be used with PostgreSQL only when explicitly configured for a single Hub or externally shared filesystem; docs must state that it does not by itself make multi-Hub artifact access safe.

### Object Backend

Add an S3-compatible object-storage backend suitable for PostgreSQL plus multi-Hub deployments:

- Required config:
  - `PANDAR_ARTIFACT_STORAGE=s3`
  - `PANDAR_ARTIFACT_S3_BUCKET`
  - `PANDAR_ARTIFACT_S3_REGION`
  - `PANDAR_ARTIFACT_S3_ENDPOINT` for S3-compatible non-AWS deployments
  - `PANDAR_ARTIFACT_S3_ACCESS_KEY_ID`
  - `PANDAR_ARTIFACT_S3_SECRET_ACCESS_KEY`
  - optional `PANDAR_ARTIFACT_S3_FORCE_PATH_STYLE=true|false`
- Object keys must be tenant/artifact scoped and generated by the hub.
- Writes must fail before job metadata is committed if object upload fails.
- Repository failure after a successful object write must best-effort delete the object and log a redacted full cause chain on cleanup failure.
- Deletes are idempotent: missing object is success.
- Object backend readiness must prove the hub can reach the configured bucket sufficiently to fail fast on bad config. Do not write permanent probe objects unless the implementation also removes them best effort.

Do not expose object-store credentials to browsers, plugins, or agents in Phase 25. Direct-to-object-store browser upload can be revisited later; this phase should keep authorization at Hub boundaries.

## Agent Artifact Retrieval

Multi-Hub object storage requires agents to fetch artifacts through Hub, not from a shared local path. Add a Hub-authenticated artifact download path for agents and carry enough command metadata for the agent to use it:

- Add `PANDAR_HUB_API_URL` to `pandar-agent`, defaulting by derivation from `PANDAR_HUB_GRPC_URL` only for the common `http(s)://host:port` case. Operators can set it explicitly when gRPC and HTTP are exposed through different hostnames or ports.
- Add `GET /api/v1/agents/{agent_id}/artifacts/{artifact_id}`.
- Authorize with `Authorization: Bearer <PANDAR_AGENT_CREDENTIAL>` plus path `agent_id`; validate the credential against the same tenant/agent credential repository used by reverse gRPC. The route must return `401` for invalid credentials, `403` for a valid agent requesting another agent's artifact, and `404` for missing artifacts without disclosing cross-tenant existence.
- The route derives tenant ownership from the authenticated agent record, not from a browser-supplied tenant id.
- The agent must not receive object-store credentials or signed object-store URLs.
- The endpoint streams artifact bytes from the configured storage backend and validates ownership by joining `job_artifacts.id = jobs.artifact_id` with `jobs.tenant_id = authenticated_agent.tenant_id` and `jobs.agent_id = authenticated_agent.id`. Any agent in the same tenant that is not assigned to a job using that artifact must be denied. Reprint/duplicate jobs keep this authorization valid because they create job rows for the target agent that reference the same artifact id.
- Extend `PrintProjectFile` with a required-for-new-commands `artifact_download_path`, for example `/api/v1/agents/{agent_id}/artifacts/{artifact_id}`. New create, retry, duplicate, reprint, and plugin print commands must include this path.
- Command handling precedence:
  - If `artifact_download_path` is present, the agent fetches from Hub and ignores `storage_path` for byte access.
  - If `artifact_download_path` is absent and `storage_path` is present, the agent uses the local filesystem reader only for commands created before this phase or tests that explicitly exercise the filesystem reader.
  - If both are absent, the agent rejects the command after printer validation with an artifact reference error.
- Object-backed commands must always include `artifact_download_path`. Dispatching an object-backed command without it is a hub bug and must fail before the command is sent or be reported as an internal command conversion error.
- Agent print dispatch must read from the Hub artifact endpoint when the command carries the new reference; local filesystem reading is transitional and must not be used to satisfy the PostgreSQL plus NATS object-storage exit criteria.
- Agent artifact fetch failures must be reported as print command failures with redacted cause chains. Do not leak local paths, object keys, bearer tokens, or agent credentials.

The preferred end state is a single agent artifact reader that fetches from Hub for new commands across both storage backends. If implementation risk requires a transitional local filesystem reader for existing tests, the transition must not weaken multi-Hub object-storage behavior.

## Upload Transport

Replace base64 job submission with streaming HTTP upload paths while preserving existing error labels and validation:

- Browser operator flow should submit `multipart/form-data` to a Next.js route handler that streams the request to `pandar-hub` with the existing server-side `apiHeaders` auth boundary. Do not send hub bearer credentials to browser JavaScript, and do not require browser-to-hub CORS in Phase 25.
- Plugin print submission should stream multipart data to the hub route instead of reading the full artifact into a base64 JSON body.
- Hub routes must accept the same logical fields as today: printer id, filename/content type, plate id, print flags, and material mappings. Browser and plugin callers may use distinct route URLs, but both must reach the same route-level validation and job creation helper.
- Enable Axum multipart handling explicitly and replace the current base64-sized request body limit with streaming size enforcement based on `PANDAR_MAX_ARTIFACT_BYTES`.
- Existing stable error labels must remain stable where semantics are unchanged:
  - `bad_request`
  - `invalid_printer_id`
  - `printer_not_found`
  - `artifact_empty`
  - `artifact_too_large`
  - `artifact_invalid_plate`
  - material mapping validation labels already in use
- Add `artifact_invalid_upload` for malformed multipart bodies, missing file parts, invalid file part metadata, and upload stream parse failures unless a more specific existing label applies.
- `artifact_invalid_base64` is retired with the JSON/base64 routes because malformed base64 no longer exists in the public upload contract. Update plugin and frontend error-code allowlists to remove `artifact_invalid_base64` and include `artifact_invalid_upload`.
- Hub, not browser/plugin, decides the storage key and stored filename.
- Upload size enforcement must work while streaming and must not depend on decoding the whole artifact into memory.
- Large-upload failures should return actionable stable labels to clients and log full redacted cause chains for operators.

The existing JSON/base64 routes may be replaced rather than kept as legacy fallback. If route shape changes for browser/plugin callers, update their tests and docs in the same milestone.

## Create, Retry, Duplicate, Reprint, And Plugin Print Behavior

- New print-job creation writes artifact bytes through `ArtifactStorage` before inserting artifact/job/command metadata.
- If metadata insertion fails after storage write, remove the stored artifact with the same backend boundary.
- Retry dispatch, duplicate, and reprint continue to reuse stored artifact metadata and must not copy artifact bytes or accept storage paths from callers.
- Plugin print uses the same storage and command creation path as browser print creation after authentication.
- Audit metadata must continue excluding artifact storage paths and sensitive transport material.

## Cleanup, Backup, Readiness, And Metrics

- Cleanup must delete artifact bytes through `ArtifactStorage` before deleting artifact rows, preserving the current retry behavior when byte deletion fails.
- Cleanup dry-run may continue reporting counts and bytes, but should not expose raw storage keys in public output.
- `/readyz` should expose an `artifact_storage` check instead of `spool`; Prometheus should use the same label. If backwards-compatible metric continuity is not kept, docs must call out the label change.
- Metrics/logs should distinguish upload validation, storage write, storage read, storage delete, and agent artifact fetch failures without leaking storage paths or tokens.
- Backup/restore docs must explain:
  - SQLite/filesystem: database plus spool directory are one backup set.
  - PostgreSQL/object storage: database backup and bucket/object lifecycle must be coordinated.
  - PostgreSQL plus NATS scaling requires object storage or an explicitly shared filesystem with known limitations.

## Configuration Contract

Minimum configuration:

- `PANDAR_ARTIFACT_STORAGE=filesystem|s3`
- `PANDAR_SPOOL_DIR` for filesystem storage
- `PANDAR_MAX_ARTIFACT_BYTES`
- `PANDAR_ARTIFACT_S3_BUCKET`
- `PANDAR_ARTIFACT_S3_REGION`
- `PANDAR_ARTIFACT_S3_ENDPOINT`
- `PANDAR_ARTIFACT_S3_ACCESS_KEY_ID`
- `PANDAR_ARTIFACT_S3_SECRET_ACCESS_KEY`
- `PANDAR_ARTIFACT_S3_FORCE_PATH_STYLE`
- agent-side `PANDAR_HUB_API_URL` when the HTTP API URL cannot be derived from `PANDAR_HUB_GRPC_URL`

Default behavior:

- SQLite defaults to filesystem storage.
- PostgreSQL single-Hub may use filesystem storage.
- PostgreSQL plus `PANDAR_CONTROL_PLANE=nats` must fail readiness when `PANDAR_ARTIFACT_STORAGE=filesystem` unless `PANDAR_ARTIFACT_FILESYSTEM_SHARED=true` is explicitly set. The explicit shared-filesystem override keeps the deployment honest but does not satisfy the Phase 25 object-storage exit criterion.
- SQLite with `PANDAR_ARTIFACT_STORAGE=s3` is allowed only if the same repository/query behavior remains backend-neutral; it is not the default and must be documented as an advanced configuration.

## Non-Goals

- No slicer artifact parsing.
- No direct browser-to-object-store presigned upload.
- No deduplication, content-addressing, checksums, malware scanning, or artifact preview generation.
- No cross-tenant artifact sharing.
- No broad rewrite of job repository behavior beyond the storage/upload boundaries.
- No legacy compatibility layer for base64 submission if browser and plugin callers are updated in this phase.

## Milestones

1. Add `ArtifactStorage` and move the current filesystem spool behind it with tests for empty/oversized writes, sanitized filenames, safe delete, readiness, and redacted errors.
2. Add streaming multipart Hub routes for browser and plugin print creation, sharing the existing validation and job creation flow.
3. Update frontend and network plugin clients to use multipart upload and keep stable error-code behavior.
4. Add agent artifact fetch through Hub and extend print commands so object-backed artifacts can be printed by any connected agent without shared POSIX storage.
5. Add the S3-compatible storage backend and object-storage readiness/delete tests. If dependency choice is non-trivial, evaluate the Rust S3 client before implementation and record the rationale in the plan.
6. Update the cleanup CLI to delete through `ArtifactStorage`, then update metrics/readiness, backup/restore docs, Docker Compose docs, and roadmap.

## Acceptance Criteria

- PostgreSQL plus NATS with object storage can create and dispatch print jobs from arbitrary Hub replicas without a shared spool directory.
- Filesystem storage remains the default simple path for SQLite/single-node deployments.
- Browser and plugin print submission no longer base64-encode artifacts through JSON request bodies.
- Hub enforces artifact size limits during upload and returns stable labels for empty, oversized, invalid printer, missing printer, and invalid plate failures.
- Agent print dispatch can fetch object-backed artifacts through Hub using agent-scoped authorization.
- Cleanup deletes artifact bytes through the selected backend before deleting rows and leaves rows for retry on delete failure.
- Reprint, duplicate, retry, plugin submission, audit, and job response behavior remain consistent across storage backends.
- Readiness/metrics expose artifact-storage health rather than assuming local spool access.
- Docs describe backup/restore and deployment differences for filesystem versus object storage.
- A scripted or automated integration scenario proves: two Hub instances share PostgreSQL, NATS, and object storage; a print job created through one Hub produces a command consumed through another Hub; the agent fetches the artifact through Hub HTTP without a shared spool directory.

## Verification Plan

- Unit tests for filesystem storage behavior and object-key safety.
- Route tests for multipart browser job creation and plugin print creation, including empty, oversized, bad plate, invalid printer id, missing printer, and repository-failure cleanup.
- Repository parity tests stay green for SQLite and PostgreSQL where metadata behavior is database-dependent.
- Agent tests for Hub artifact fetch success, redacted fetch failure, and local path rejection for filesystem-only reader paths that remain.
- Cleanup tests prove byte deletion happens before row deletion and delete failures preserve artifact rows.
- Readiness/metrics tests prove the new `artifact_storage` check and labels.
- Plugin boundary tests prove multipart request shape and stable error mapping without leaking artifact paths.
- Frontend build verifies the browser upload path compiles after removing base64 server-action submission.
- Add an integration smoke script or test harness for PostgreSQL plus NATS plus object storage plus two Hub instances. It may use a local S3-compatible service or repository fake, but it must exercise cross-Hub command dispatch and Hub-mediated agent artifact fetch rather than a shared filesystem.
- Final verification: `cargo fmt --check`, `cargo clippy --workspace`, plugin tests, hub route/repository tests, `cargo nextest run --manifest-path "Cargo.toml" --workspace`, and `npm --prefix frontend run build`.
