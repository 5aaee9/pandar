# Phase 11 Provisioning And Admin Boundaries Design

## Context

Pandar currently has tenant-scoped API-token and Clerk/Logto JWT authentication for operational APIs, but several development-era ergonomics remain:

- `GET /api/v1/summary`, `GET /api/v1/tenants`, and `POST /api/v1/tenants` are unauthenticated cross-tenant endpoints.
- Tenant admins can create agents, jobs, and printer refresh commands, but they cannot manage local users, service tokens, or external identity links through HTTP APIs.
- Fresh deployments still need repository/test fixtures or direct database access to create the first tenant admin and API token.
- API tokens cannot be revoked.
- Agent setup still requires manually copying persistent tenant and agent IDs from separate API responses.

Phase 11 removes those gaps while keeping the existing Rust-owned tenant membership model. Clerk and Logto remain authentication providers only; Pandar remains authoritative for tenant users, roles, service tokens, and audit events.

## Goals

1. Add a bootstrap/global authorization boundary for cross-tenant administrative endpoints.
2. Add an API bootstrap flow that creates a tenant, tenant admin user, and first API token without direct database fixtures.
3. Protect cross-tenant summary and tenant listing/creation so ordinary tenant users cannot enumerate other tenants.
4. Add tenant-admin HTTP APIs for local user creation/listing, external identity linking/listing, API-token creation/listing/revocation, and agent pairing bundle creation.
5. Persist token revocation and exclude revoked API tokens from bearer authentication.
6. Audit provisioning actions: bootstrap tenant admin creation, tenant creation, user creation, external identity linking, API-token creation/revocation, role changes when introduced by user-management endpoints, and agent pairing bundle creation.
7. Document a fresh-deployment bootstrap flow and the agent pairing/token-rotation boundary.

## Non-Goals

- No frontend admin UI in Phase 11. The phase provides HTTP APIs and documentation.
- No provider organization synchronization. Clerk/Logto organization membership is still ignored for Pandar authorization.
- No email delivery, invitation acceptance pages, or magic-link workflow. "Invite/linking" means tenant admins can create a pending local user and bind a verified provider subject to that user.
- No cryptographic agent registration enforcement on the existing gRPC reverse connection in Phase 11. This phase returns a pairing bundle that removes hand-copying IDs and documents the future token-rotation protocol; enforcing agent pairing secrets belongs with a protobuf/gRPC auth phase.
- No SeaORM expansion beyond changes needed for Phase 11. Phase 12 owns completing the repository migration.

## Design Decisions

### Bootstrap Authority

`pandar-hub` gains an optional bootstrap/global admin token configured by `PANDAR_BOOTSTRAP_TOKEN`.

- Requests authenticate this authority with `Authorization: Bearer <PANDAR_BOOTSTRAP_TOKEN>`.
- Bootstrap-authorized endpoints do not accept tenant API tokens or Clerk/Logto JWTs as substitutes.
- If the endpoint requires bootstrap authority:
  - missing `Authorization` returns `401 {"error":"missing_auth_token"}`;
  - non-Bearer or wrong token returns `401 {"error":"invalid_auth_token"}`;
  - an unset or empty `PANDAR_BOOTSTRAP_TOKEN` makes bootstrap-only endpoints unavailable and returns `403 {"error":"bootstrap_disabled"}` after a Bearer token is supplied.
- The bootstrap token is not stored in the database and is never returned by APIs.

The bootstrap authority protects:

- `GET /api/v1/summary`
- `GET /api/v1/tenants`
- `POST /api/v1/tenants`
- `POST /api/v1/bootstrap/tenant-admin`

This gives Pandar an explicit cross-tenant authority without adding a global-user table.

### Bootstrap Tenant Admin API

`POST /api/v1/bootstrap/tenant-admin` creates a tenant, a local tenant admin user, and a first API token in one request.

Request:

```json
{
  "tenant_slug": "acme",
  "tenant_display_name": "Acme",
  "admin_email": "admin@example.com",
  "admin_display_name": "Admin",
  "api_token_name": "bootstrap-admin"
}
```

Response status: `201 Created`.

Response:

```json
{
  "tenant": { "id": "...", "slug": "acme", "display_name": "Acme", "created_at": "..." },
  "user": {
    "id": "...",
    "tenant_id": "...",
    "email": "admin@example.com",
    "display_name": "Admin",
    "role": "tenant_admin",
    "created_at": "..."
  },
  "api_token": {
    "id": "...",
    "tenant_id": "...",
    "user_id": "...",
    "name": "bootstrap-admin",
    "token": "pandar_...",
    "created_at": "...",
    "last_used_at": null,
    "revoked_at": null
  }
}
```

The plaintext API token is generated by the hub, returned once, and stored only as the existing SHA-256 token hash. The generated token uses a `pandar_` prefix plus a UUID-derived random value to avoid adding a new dependency.

The endpoint may be used for the first deployment tenant and for later bootstrap-token-created tenants. Duplicate slug and duplicate token-name behavior uses existing repository conflict errors.

The tenant, admin user, first API token, and bootstrap audit events must be created atomically. If any later step fails after the tenant insert begins, the entire bootstrap operation rolls back and no partial tenant, user, token, or audit events remain. The implementation may add a repository-level bootstrap transaction instead of composing existing independent repository calls at the route layer.

`POST /api/v1/bootstrap/tenant-admin` emits these audit events in the new tenant transaction:

- `tenant.bootstrap` with `actor_type = "bootstrap"`, `target_type = "tenant"`, `target_id = tenant id`, and metadata containing `tenant_slug`;
- `user.create` with `actor_type = "bootstrap"`, `target_type = "user"`, `target_id = admin user id`, and metadata containing `email` and `role = "tenant_admin"`;
- `api_token.create` with `actor_type = "bootstrap"`, `target_type = "api_token"`, `target_id = token id`, and metadata containing `name` and `user_id`.

### Cross-Tenant Tenant APIs

`POST /api/v1/tenants` remains available only to the bootstrap authority. It creates a tenant without creating an admin user and records a `tenant.create` audit event scoped to the created tenant with `actor_type = "bootstrap"`.

`GET /api/v1/summary` and `GET /api/v1/tenants` require bootstrap authority. A tenant admin/operator/viewer API token must not list or summarize other tenants.

### Tenant User Management APIs

Tenant admins get these routes:

- `GET /api/v1/tenants/{tenant_id}/users`
- `POST /api/v1/tenants/{tenant_id}/users`
- `PATCH /api/v1/tenants/{tenant_id}/users/{user_id}/role`

All require `UserRole::TenantAdmin` for the path tenant.

`POST /users` request:

```json
{
  "email": "operator@example.com",
  "display_name": "Operator",
  "role": "operator"
}
```

`PATCH /role` request:

```json
{ "role": "viewer" }
```

Supported role strings are exactly `tenant_admin`, `operator`, and `viewer`.

Responses:

- `GET /users` returns `200 OK` with `{ "users": [UserResponse...] }`.
- `POST /users` returns `201 Created` with a `UserResponse`.
- `PATCH /role` returns `200 OK` with the updated `UserResponse`.

`UserResponse`:

```json
{
  "id": "...",
  "tenant_id": "...",
  "email": "operator@example.com",
  "display_name": "Operator",
  "role": "operator",
  "created_at": "..."
}
```

Repository additions:

- `list_users_for_tenant(tenant_id)`
- `update_user_role(tenant_id, user_id, role)`

Audit actions:

- `user.create`
- `user.role_update`

User email is unique per tenant. Creating a duplicate `(tenant_id, email)` returns `409 {"error":"user_email_exists"}`. Duplicate detection is represented as a repository error so SQLite and PostgreSQL map to the same API behavior.

### External Identity Linking APIs

Tenant admins get these routes:

- `GET /api/v1/tenants/{tenant_id}/users/{user_id}/identities`
- `POST /api/v1/tenants/{tenant_id}/users/{user_id}/identities`

`POST /identities` request:

```json
{
  "provider": "clerk",
  "subject": "user_..."
}
```

The API uses the existing `link_external_identity` rules:

- the local user must belong to the path tenant;
- `(tenant_id, provider, subject)` is unique;
- `(tenant_id, user_id, provider)` is unique.

Repository addition:

- `list_external_identities_for_user(tenant_id, user_id)`

Audit action:

- `user_identity.link`

Responses:

- `GET /identities` returns `200 OK` with `{ "identities": [UserIdentityResponse...] }`.
- `POST /identities` returns `201 Created` with a `UserIdentityResponse`.

`UserIdentityResponse`:

```json
{
  "id": "...",
  "tenant_id": "...",
  "user_id": "...",
  "provider": "clerk",
  "subject": "user_...",
  "created_at": "..."
}
```

### API Token Management APIs

Tenant admins get these routes:

- `GET /api/v1/tenants/{tenant_id}/users/{user_id}/api-tokens`
- `POST /api/v1/tenants/{tenant_id}/users/{user_id}/api-tokens`
- `DELETE /api/v1/tenants/{tenant_id}/api-tokens/{token_id}`

`POST /api-tokens` request:

```json
{ "name": "automation" }
```

The create response includes the plaintext generated token once. List and revoke responses never include plaintext tokens or token hashes.

`api_tokens` gains `revoked_at TEXT NULL` in both SQLite and PostgreSQL migrations.

Repository additions:

- `list_api_tokens_for_user(tenant_id, user_id)`
- `revoke_api_token(tenant_id, token_id)`

Authentication changes:

- `authenticate_bearer` only matches API tokens where `revoked_at IS NULL`.
- Revoking an unknown token returns `404 {"error":"api_token_not_found"}`.
- Revoking an already revoked token is idempotent and returns the token metadata with its existing `revoked_at`.

Audit actions:

- `api_token.create`
- `api_token.revoke`

Responses:

- `GET /api-tokens` returns `200 OK` with `{ "api_tokens": [ApiTokenResponse...] }`.
- `POST /api-tokens` returns `201 Created` with an `ApiTokenWithPlaintextResponse`.
- `DELETE /api-tokens/{token_id}` returns `200 OK` with an `ApiTokenResponse`.

`ApiTokenResponse`:

```json
{
  "id": "...",
  "tenant_id": "...",
  "user_id": "...",
  "name": "automation",
  "created_at": "...",
  "last_used_at": null,
  "revoked_at": null
}
```

`ApiTokenWithPlaintextResponse` has the same fields plus a one-time `token` field:

```json
{
  "id": "...",
  "tenant_id": "...",
  "user_id": "...",
  "name": "automation",
  "token": "pandar_...",
  "created_at": "...",
  "last_used_at": null,
  "revoked_at": null
}
```

### Agent Pairing Bundle API

Tenant admins get:

- `POST /api/v1/tenants/{tenant_id}/agent-pairings`

Request:

```json
{ "name": "workshop-agent" }
```

Response status: `201 Created`.

Response:

```json
{
  "agent": {
    "id": "...",
    "tenant_id": "...",
    "name": "workshop-agent",
    "status": "offline",
    "created_at": "..."
  },
  "agent_env": "PANDAR_TENANT_ID=...\nPANDAR_AGENT_ID=...\nPANDAR_AGENT_NAME=workshop-agent\n"
}
```

This endpoint wraps agent creation and returns a copy/paste deployment bundle, so operators no longer need to discover and copy persistent database IDs by hand. The existing `POST /api/v1/tenants/{tenant_id}/agents` route remains for compatibility with existing tests and scripts, but the documented setup path uses `agent-pairings`.

Audit action:

- `agent.pairing_bundle`

`POST /agent-pairings` returns `201 Created`. Empty names return `400 {"error":"bad_request"}` and duplicate agent names use the existing `409 {"error":"agent_name_exists"}` behavior.

The documented future token-rotation protocol is:

1. tenant admin creates or rotates a short-lived agent pairing secret;
2. agent presents the secret in its reverse gRPC hello metadata;
3. hub binds or rotates the persisted agent credential and invalidates the previous secret;
4. subsequent gRPC connections authenticate with the rotated agent credential.

Phase 11 documents this flow and keeps runtime gRPC enforcement unchanged.

### Audit Metadata

Audit events remain tenant-scoped. Bootstrap actions record `actor_type = "bootstrap"` and no `user_id`. Tenant-admin actions record `actor_type = "user"` and the acting user id.

Metadata is compact JSON with stable fields needed for operator review:

- user creation: created email and role;
- role update: previous and new role;
- identity link: provider and subject;
- token create/revoke: token name and affected user id;
- pairing bundle: agent name.

### Error Handling

Boundary validation applies only to incoming API payloads:

- empty required strings return `400 {"error":"bad_request"}`;
- invalid roles return `400 {"error":"invalid_user_role"}`;
- invalid tenant ids keep `400 {"error":"invalid_tenant_id"}`;
- missing user/token/tenant repository results map to `404` codes.

Repository/database errors keep their cause chain in logs through the existing `{err:#}` style.

### Frontend Impact

No admin UI is added.

The existing dashboard must not depend on unauthenticated cross-tenant endpoints after this phase. When `APP_TENANT_ID` is configured, server-side frontend reads should use tenant-scoped endpoints and avoid treating protected `/summary` or `/tenants` failures as tenant-dashboard failures. When `APP_TENANT_ID` is not configured, the dashboard may still use `/summary` and `/tenants`, but deployments must provide a bootstrap/global bearer through existing frontend bearer forwarding.

## Acceptance Criteria

1. A fresh SQLite deployment can create a tenant, tenant admin, and first API token through `POST /api/v1/bootstrap/tenant-admin` without direct database fixtures.
2. A fresh PostgreSQL deployment can create a tenant, tenant admin, and first API token through the same bootstrap transaction path when `PANDAR_TEST_POSTGRES_URL` is configured.
3. Bootstrap tenant admin creation is atomic: forced failures in user/token/audit creation leave no partial tenant provisioning artifacts.
4. `GET /api/v1/summary`, `GET /api/v1/tenants`, and `POST /api/v1/tenants` reject missing, invalid, or tenant-scoped bearer tokens and accept only the configured bootstrap token.
5. A tenant admin can list/create users, update user roles, link/list Clerk or Logto subjects, create/list/revoke API tokens, and create an agent pairing bundle only inside its tenant.
6. Operators and viewers cannot call tenant-admin provisioning endpoints.
7. Duplicate tenant-local user email returns `409 {"error":"user_email_exists"}` for SQLite and PostgreSQL.
8. Revoked API tokens can no longer authenticate, while non-revoked API tokens and linked external identities keep their existing behavior.
9. Provisioning actions are visible in tenant audit events with the action names listed in this spec.
10. SQLite and PostgreSQL migrations stay equivalent for `api_tokens.revoked_at`.
11. Existing tenant operational APIs continue to authorize with API tokens and external identities.
12. Tests cover bootstrap auth, cross-tenant denial, tenant-admin provisioning, token revocation, audit events, and repository behavior for both database backends where backend-dependent.
